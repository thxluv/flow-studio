## Деплой вручную (самое частое)

### 1. Залить код на GitHub + Render

1. Двойной клик **`deploy.bat`** в корне проекта
2. Enter (или напиши сообщение коммита)
3. Подожди 1–2 мин (GitHub Pages) и 3–5 мин (Render)

**Локально FlowPhoto:** двойной клик **`start-flowphoto.bat`** → http://127.0.0.1:8000/

### 2. Только перезапуск Render (без кода)

Render → **flowphoto** → **Manual Deploy** → **Deploy latest commit**

Полезно: сразу сделать бэкап БД в Storj, если на сервере уже есть фото.

### 3. Переменные окружения Render

Render → **flowphoto** → **Environment**

- **Импорт:** **Add from .env** → выбери `archive/render-flowphoto.env` (сначала замени секреты в файле)
- **Экспорт (бэкап настроек):** **Export Env Vars**

Шаблон env: `archive/render-flowphoto.env`  
Подробно про Storj: `archive/STORJ-FREE-RENDER.md`

## Как следить, что всё живо

### Health (главная проверка)

https://flowphoto.onrender.com/health

| Поле | Норма |
|------|-------|
| `status` | `"ok"` |
| `backup_configured` | `true` (Storj настроен) |
| `version` | **3.5.0** |
| `storage.photos_count` | сколько фото на сервере |
| `storage.vaults_count` | сколько аккаунтов FlowVault (должно быть ≥1 если сейф создан) |

### Логи Render

Render → **flowphoto** → **Logs**

Ищи строки:
- `Backup uploaded` — БД ушла в Storj
- `Restore` — подтянули БД после рестарта
- ошибки `403` — проблема с ключами Storj

### Storj

Buckets → `flowphoto-backup` → папка `flowphoto/`

Файлы: `flowphoto_YYYYMMDD_HHMMSS.db`  
**Автобэкап (v3.4.3+):** через **~60 сек** после загрузки фото или создания FlowVault.  
Плюс плановый бэкап **каждые 30 мин** (`FLOWPHOTO_BACKUP_INTERVAL=1800`).

Проверка в `/health` → `backup.auto_pending: true` (ждёт заливку), `backup.last_upload_at` (когда ушло в Storj).

## Два вида бэкапа (не путать)

| Тип | Где | Что сохраняет |
|-----|-----|----------------|
| **Storj** (сервер) | Облако | Вся база `flowphoto.db` — все фото на Render |
| **`.flowvault`** (браузер) | Твой телефон/ПК | Ключи расшифровки и права загрузки |

**FlowVault — два пароля:**
1. **Пароль бэкапа** — только для файла `.flowvault`
2. **Пароль FlowVault** — вход в сейф (при «Создать»). Без него в аккаунт не зайти, даже после восстановления файла.

## Что где менять

| Задача | Файл / место |
|--------|----------------|
| Ссылка FlowPhoto в FlowNote | `flow-config.js`, `public-config.json` |
| Секрет сейфа (прод) | Render → `FLOWPHOTO_VAULT_SECRET` (не в git!) |
| Storj ключи | Render → `FLOWPHOTO_BACKUP_*` |
| Интервал бэкап БД | Render → `FLOWPHOTO_BACKUP_INTERVAL` (1800 = 30 мин) |
| Лимит диска сервера | Render → `FLOWPHOTO_MAX_STORAGE_BYTES` |
| UI FlowPhoto / FlowVault | `flowphoto-server/templates/`, `static/js/` |
| FlowNote интерфейс | `index.html` |

## Типичные проблемы

| Симптом | Что сделать |
|---------|-------------|
| FlowPhoto «спит» 30+ сек | Free tier — первый запрос будит сервер |
| Фото пропали после рестарта | Проверь Storj: есть ли `.db`? `backup_configured: true`? |
| Аккаунт Vault «не найден» после деплоя | `/health` → `vaults_count: 0`? В Storj мог залиться пустой бэкап (исправлено в v3.4.2+). Создай сейф заново, залей фото, дождись бэкапа с `vaults≥1` |
| `.flowvault` серый на телефоне | Обнови сайт (v3.4.1+) — выбор любого файла, не только по расширению |
| Не входит в Vault после бэкапа | Нужен **пароль FlowVault**, не пароль бэкапа |
| Render не деплоится | GitHub push прошёл? Render → Events → ошибка сборки |

### Лицензия на код (заметка для юриста / себя)

- **Зачем:** если код публичный на GitHub без LICENSE — по умолчанию **все права у автора**, но другие формально не могут легально использовать/копировать с ясными правилами; с публичным repo есть риск форков.
- **Цель владельца:** код **не открытый**, **своя ИС** — тогда нужна **проприетарная** модель: репозиторий private, в документах «все права защищены», без open-source лицензии.
- **MIT/Apache** — только если сознательно разрешаем копирование; **нам не подходит** при желании скрыть код.

### Логи Render (для юриста)

- Приложение **не пишет IP в SQLite** долгосрочно (только rate-limit в памяти).
- **Render** как хостинг может вести **свои access-логи** (IP, URL, время) — срок хранения уточнить в политике Render / у юриста для 152-ФЗ.
